# نوبتینو — سامانه رزرو آنلاین خدمات زیبایی

نوبتینو یک پروژه Django برای راه‌اندازی مارکت‌پلیس رزرو خدمات زیبایی است. ایده اصلی پروژه این است که کاربر ابتدا با دسته‌های کلی خدمات مثل ناخن، مژه، پوست و مو یا ماساژ آشنا می‌شود، سپس شهر خود را انتخاب می‌کند و فقط ارائه‌دهنده‌های همان شهر را می‌بیند. ارائه‌دهنده‌ها هم صفحه اختصاصی، مدیریت خدمات، مدیریت رزروها و اشتراک قابل تمدید دارند.

## ویژگی‌های اصلی

- ورود کاربر با شماره موبایل و OTP.
- صفحه اصلی عمومی و city-agnostic برای نمایش کلی خدمات، جستجو، بلاگ و متخصص‌های منتخب.
- انتخاب دسته خدمت و سپس انتخاب/ذخیره شهر برای نمایش providerهای همان شهر.
- صفحه عمومی اختصاصی برای هر provider با سرویس‌ها، قیمت، زمان و اطلاعات تماس.
- رزرو وقت برای یک سرویس مشخص از یک provider.
- داشبورد مشتری برای مشاهده رزروها.
- تبدیل کاربر عادی به provider با ثبت اطلاعات اولیه و انتخاب پلن.
- بررسی زنده slug برای provider با حداقل طول بیشتر از ۴ کاراکتر و یکتا بودن.
- داشبورد provider برای مدیریت ظاهر صفحه، رنگ‌ها، شهر، آدرس، خدمات و رزروها.
- ساختار خدمات ادمین‌محور: provider فقط از سرویس‌هایی انتخاب می‌کند که ادمین از قبل تعریف کرده است.
- سیستم اشتراک با روزشمار، تمدید و قابلیت auto-renew.
- بلاگ قابل مدیریت از پنل Django Admin برای ادمین‌های سایت.
- محدودسازی پنل ادمین برای `is_staff` و نمایش کامل برای `is_superuser`.

## نصب و راه‌اندازی

ابتدا وارد مسیر پروژه شوید:

```bash
cd /workspace/appointment
```

یک محیط مجازی بسازید و فعال کنید:

```bash
python -m venv .venv
source .venv/bin/activate
```

وابستگی‌ها را نصب کنید:

```bash
pip install -r requirements.txt
```

مایگریشن‌ها را اجرا کنید:

```bash
python manage.py migrate
```

یک superuser بسازید:

```bash
python manage.py createsuperuser
```

سرور توسعه را اجرا کنید:

```bash
python manage.py runserver
```

آدرس‌های مهم:

- صفحه اصلی: `http://127.0.0.1:8000/`
- ورود با OTP: `http://127.0.0.1:8000/accounts/`
- پنل ادمین: `http://127.0.0.1:8000/admin/`

## جریان کار کاربران

### کاربر عادی

1. وارد سایت می‌شود و صفحه اصلی عمومی را می‌بیند.
2. می‌تواند در سرچ دنبال شهر، provider یا سرویس بگردد.
3. یک دسته خدمت مثل ناخن را انتخاب می‌کند.
4. شهر را وارد یا تغییر می‌دهد.
5. لیست providerهای همان شهر و همان دسته خدمت را می‌بیند.
6. وارد صفحه provider می‌شود.
7. سرویس موردنظر را انتخاب و زمان رزرو می‌کند.
8. در داشبورد مشتری رزروهای خود را می‌بیند.

### ارائه‌دهنده خدمات

1. ابتدا مثل کاربر عادی با شماره موبایل وارد می‌شود.
2. روی گزینه تبدیل به provider می‌زند.
3. نام، شهر، slug اختصاصی، توضیحات و اطلاعات تماس را وارد می‌کند.
4. سیستم به صورت زنده بررسی می‌کند slug قابل استفاده هست یا نه.
5. اطلاعات اولیه ذخیره می‌شود؛ حتی اگر هنوز پلن نخریده باشد.
6. به صفحه پلن‌ها منتقل می‌شود و یک پلن انتخاب می‌کند.
7. در داشبورد provider می‌تواند ظاهر صفحه، رنگ‌ها، آدرس، شهر و خدمات را مدیریت کند.

## مدل‌های مهم خدمات و محصولات

این بخش مهم‌ترین بخش معماری پروژه است؛ چون تفاوت بین «دسته خدمت»، «قالب خدمت»، «سرویس provider» و «محصول» باید واضح باشد.

### ServiceCategory

`ServiceCategory` دسته‌بندی اصلی خدمات است که فقط ادمین آن را تعریف می‌کند.

مثال‌ها:

- ناخن
- مژه
- پوست و مو
- ماساژ
- آرایش

هدف این مدل این است که صفحه اصلی و مسیر کشف خدمات با دسته‌های کلی و قابل کنترل ساخته شود. کاربر در صفحه اصلی دسته کلی را می‌بیند، نه سرویس‌های هر provider در همه شهرها.

فیلدهای مهم:

| فیلد | توضیح |
| --- | --- |
| `name` | نام دسته، مثل «ناخن» |
| `slug` | آدرس یکتای دسته در URL |
| `description` | توضیح کوتاه برای نمایش در UI |
| `icon` | آیکن متنی/ایموجی برای کارت دسته |
| `image` | تصویر اختیاری دسته |
| `is_active` | فعال یا غیرفعال بودن دسته |
| `sort_order` | ترتیب نمایش |

نمونه کاربرد:

> ادمین دسته «ناخن» را می‌سازد تا کاربران بتوانند وارد صفحه دسته ناخن شوند و پس از انتخاب شهر، ناخن‌کارهای همان شهر را ببینند.

---

### ServiceTemplate

`ServiceTemplate` قالب یا سرویس استانداردی است که زیرمجموعه یک `ServiceCategory` قرار می‌گیرد و توسط ادمین ساخته می‌شود.

مثال برای دسته «ناخن»:

- کاشت ناخن
- ترمیم ناخن
- ژلیش
- مانیکور

مثال برای دسته «مژه»:

- اکستنشن مژه
- لیفت مژه
- لمینت مژه

هدف این مدل جلوگیری از این است که provider هر چیزی را به عنوان سرویس وارد کند. provider فقط می‌تواند از `ServiceTemplate`های فعال انتخاب کند.

فیلدهای مهم:

| فیلد | توضیح |
| --- | --- |
| `category` | دسته اصلی که این قالب به آن تعلق دارد |
| `name` | نام سرویس استاندارد |
| `description` | توضیح پیش‌فرض یا راهنمای سرویس |
| `suggested_duration` | زمان پیشنهادی انجام سرویس به دقیقه |
| `is_active` | فعال/غیرفعال بودن این قالب |

نمونه کاربرد:

> ادمین زیر دسته «ناخن» یک template با نام «ترمیم ناخن» می‌سازد. هر provider که خدمات ناخن ارائه می‌دهد، می‌تواند همین template را انتخاب کند و فقط قیمت، زمان و توضیح اختصاصی خودش را وارد کند.

---

### Service

`Service` سرویس واقعی قابل فروش/رزرو توسط provider است. این مدل متعلق به provider است، اما نوع آن باید از روی `ServiceTemplate` انتخاب شود.

به زبان ساده:

- `ServiceTemplate` را ادمین می‌سازد.
- `Service` را provider بر اساس آن template فعال می‌کند.

مثال:

- ادمین template «ژلیش» را تعریف کرده است.
- provider شهر قم آن template را انتخاب می‌کند.
- provider قیمت را ۳۵۰٬۰۰۰ تومان، زمان را ۶۰ دقیقه و توضیح اختصاصی خودش را وارد می‌کند.
- حالا کاربر در صفحه آن provider می‌تواند «ژلیش» را رزرو کند.

فیلدهای مهم:

| فیلد | توضیح |
| --- | --- |
| `provider` | ارائه‌دهنده مالک این سرویس |
| `template` | سرویس استاندارد انتخاب‌شده از لیست ادمین |
| `name` | نام سرویس؛ در حالت عادی از template پر می‌شود |
| `description` | توضیح اختصاصی provider |
| `price` | قیمت سرویس |
| `duration` | مدت زمان انجام سرویس به دقیقه |
| `deposit_amount` | مبلغ بیعانه |
| `image` | تصویر اختیاری سرویس |
| `is_active` | فعال یا غیرفعال بودن سرویس |

نکته مهم:

اگر `deposit_amount` از `price` بیشتر باشد، مدل خطای اعتبارسنجی می‌دهد. همچنین اگر سرویس template داشته باشد و نام خالی باشد، نام از template گرفته می‌شود.

---

### ProductCategory

`ProductCategory` دسته‌بندی محصولات فیزیکی یا قابل فروش provider است. این مدل با `ServiceCategory` فرق دارد.

تفاوت مهم:

- `ServiceCategory` برای دسته‌بندی خدمات قابل رزرو است و توسط ادمین تعریف می‌شود.
- `ProductCategory` برای دسته‌بندی محصولات فروشگاهی یک provider است و به همان provider تعلق دارد.

مثال‌ها:

- محصولات مراقبت ناخن
- کرم‌ها
- شامپوها
- ابزار زیبایی

فیلدهای مهم:

| فیلد | توضیح |
| --- | --- |
| `provider` | صاحب این دسته محصول |
| `name` | نام دسته محصول |

قانون یکتایی:

هر provider نمی‌تواند دو دسته محصول با نام یکسان داشته باشد؛ اما providerهای مختلف می‌توانند دسته‌های هم‌نام داشته باشند.

---

### Product

`Product` محصول واقعی قابل فروش توسط provider است. این مدل برای فروش محصول استفاده می‌شود، نه رزرو نوبت.

مثال‌ها:

- روغن کوتیکول
- کرم دست
- شامپو تخصصی
- پکیج مراقبت بعد از کاشت ناخن

فیلدهای مهم:

| فیلد | توضیح |
| --- | --- |
| `provider` | ارائه‌دهنده/فروشنده محصول |
| `category` | دسته محصول، اختیاری |
| `name` | نام محصول |
| `description` | توضیح محصول |
| `image` | تصویر محصول |
| `price` | قیمت محصول |
| `stock` | موجودی |
| `is_active` | فعال یا غیرفعال بودن محصول |
| `discount_percent` | درصد تخفیف |
| `discount_starts_at` | شروع تخفیف |
| `discount_ends_at` | پایان تخفیف |

تفاوت `Product` با `Service`:

| مورد | Service | Product |
| --- | --- | --- |
| هدف | رزرو زمان برای انجام خدمت | فروش کالا/محصول |
| وابستگی به زمان | دارد | ندارد |
| رزرو appointment | دارد | ندارد |
| دسته‌بندی اصلی | از `ServiceTemplate` و `ServiceCategory` | از `ProductCategory` |
| تعریف نوع | کنترل‌شده توسط ادمین | متعلق به provider |

## مدیریت شهر و فیلتر ارائه‌دهنده‌ها

چون خدمات وابسته به شهر هستند، صفحه اصلی providerهای همه شهرها را به عنوان محصول نهایی نمایش نمی‌دهد. مسیر درست این است:

1. کاربر دسته خدمت را انتخاب می‌کند.
2. شهر را وارد می‌کند.
3. شهر برای کاربر ذخیره می‌شود.
4. فقط providerهای همان شهر و همان دسته نمایش داده می‌شوند.

این طراحی باعث می‌شود کاربر تهران، providerهای قم را نبیند و تجربه رزرو دقیق‌تر شود.

## ثبت‌نام ارائه‌دهنده و slug اختصاصی

هر provider یک `slug` اختصاصی دارد که در URL صفحه عمومی او استفاده می‌شود.

قوانین فعلی:

- slug باید یکتا باشد.
- slug باید بیشتر از ۴ کاراکتر باشد.
- هنگام تایپ slug، وضعیت قابل استفاده بودن به صورت زنده بررسی می‌شود.
- ثبت اولیه provider قبل از خرید پلن ذخیره می‌شود تا اگر کاربر بعدا برگشت، اطلاعاتش از بین نرفته باشد.

نمونه URL:

```text
/providers/my-beauty-studio/
```

## اشتراک و تمدید

provider پس از ثبت اولیه باید یک پلن انتخاب کند. پلن‌ها توسط ادمین ساخته می‌شوند و شامل ویژگی‌هایی مثل موارد زیر هستند:

- مدت زمان پلن
- قیمت
- امکان دریافت بیعانه
- امکان ارسال پیامک
- امکان مدیریت محصولات
- امکان مشاهده آمار
- امکان ایجاد تخفیف
- امکان برنامه وفاداری

در مدل `Subscription` موارد زیر وجود دارد:

- `start_date`: تاریخ شروع
- `end_date`: تاریخ پایان
- `is_active`: فعال بودن اشتراک
- `auto_renew`: تمدید خودکار
- `renewal_price_snapshot`: قیمت ثبت‌شده هنگام خرید/تمدید
- `days_remaining`: روزهای باقی‌مانده
- `should_show_renewal`: نمایش دکمه تمدید وقتی ۳ روز یا کمتر مانده باشد

نکته مهم:

حتی اگر پلن در ابتدا رایگان باشد، برای تمدید باید قیمت جدید پلن از دیتابیس خوانده شود. به همین دلیل قیمت پلن‌ها باید همیشه در دیتابیس مدیریت شوند.

## بلاگ و پنل ادمین

پروژه یک مدل `Blog` دارد که ادمین‌ها می‌توانند برای تولید محتوا از آن استفاده کنند.

قابلیت‌های بلاگ:

- عنوان
- slug
- محتوا
- نویسنده
- تصویر
- وضعیت انتشار
- تاریخ انتشار

برای ادمین‌های سایت که برنامه‌نویس نیستند، پنل ادمین بلاگ با fieldsetهای ساده‌تر و توضیح کاربری آماده شده است. همچنین امکان استفاده از HTML ساده در محتوا وجود دارد.

### سطح دسترسی ادمین

- `is_superuser`: دسترسی کامل به همه مدل‌ها و اپلیکیشن‌ها.
- `is_staff`: دسترسی محدود به اپلیکیشن‌های مجاز مثل بلاگ و پشتیبانی.

لیست اپ‌های مجاز staff در تنظیمات با `ADMIN_STAFF_ALLOWED_APPS` مشخص می‌شود.

## دستورات توسعه و تست

بررسی سلامت پروژه:

```bash
python manage.py check
```

ساخت مایگریشن‌های جدید:

```bash
python manage.py makemigrations
```

```bash
python manage.py makemigrations --check
```

```bash
python manage.py migrate
```

اجرای تست‌ها:

```bash
python manage.py test
```

بررسی خطاهای whitespace در diff:

```bash
git diff --check
```

## نکات توسعه آینده

برای کامل‌تر شدن پروژه، پیشنهاد می‌شود موارد زیر در مراحل بعد پیاده‌سازی شوند:

- اتصال واقعی OTP به سرویس پیامک.
- اتصال پرداخت واقعی برای خرید پلن و پرداخت بیعانه.
- پیاده‌سازی کیف پول و منطق واقعی auto-renew.
- صفحه کامل مدیریت رزروها برای provider.
- صفحه کامل مدیریت ساعات کاری provider.
- ادیتور WYSIWYG کامل‌تر برای بلاگ در پنل اختصاصی غیر از Django Admin.
- سیستم جستجوی پیشرفته با فیلتر قیمت، امتیاز، شهر و زمان آزاد.
- تست‌های واحد و integration برای رزرو، اشتراک، city filtering و slug validation.
- بهینه‌سازی SEO برای صفحات دسته خدمات، providerها و بلاگ.

## خلاصه معماری خدمات

برای جلوگیری از اشتباه در توسعه آینده، این خلاصه را همیشه در نظر بگیرید:

```text
ServiceCategory
    دسته کلی خدمات توسط ادمین
    مثال: ناخن

ServiceTemplate
    سرویس استاندارد زیرمجموعه دسته توسط ادمین
    مثال: کاشت ناخن، ترمیم، ژلیش

Service
    سرویس واقعی قابل رزرو که provider از روی ServiceTemplate فعال می‌کند
    مثال: ژلیش توسط سالن X در شهر قم با قیمت مشخص

ProductCategory
    دسته محصول متعلق به provider
    مثال: محصولات مراقبت ناخن

Product
    کالای قابل فروش provider
    مثال: روغن کوتیکول یا کرم دست
```

این تفکیک باعث می‌شود marketplace هم کنترل محتوایی داشته باشد، هم برای کاربر قابل جستجو و قابل فیلتر باشد، و هم provider بتواند قیمت و توضیحات اختصاصی خودش را مدیریت کند.
